# yanxu-html 安全模型

本库的目标是让常见的服务端 HTML 注入错误变得难以意外发生。它提供安全构建原语，不替代完整的内容安全策略、浏览器隔离或业务授权。

## 信任路径

| 输入来源 | 应使用 | 行为 |
| --- | --- | --- |
| 用户输入、数据库、接口、文件 | `文字`、文字属性 | 自动 HTML 转义。 |
| 布尔状态 | 布尔属性 | `真`输出名称，`假`省略。 |
| 已审计的常量或可信生成器 | `原始` | 原样输出，调用者承担安全责任。 |

不要把“已经手工转义的用户输入”传入`原始`。编码与上下文相关，手工拼接很容易在属性、脚本、样式或 SVG 上下文中失效。

## 名称验证

标签名必须匹配`^[A-Za-z][A-Za-z0-9:-]*$`；属性名必须匹配`^[A-Za-z_:][A-Za-z0-9_.:-]*$`。因此空白、引号、控制字符、`<`、`>`和`=`不能进入名称位置。

白名单允许自定义元素、命名空间名称和`data-*`、`aria-*`一类属性，但不判断某个名称是否属于标准 HTML。

## URL 属性

以下属性不区分大小写地进入 URL 校验：

- `href`
- `src`
- `action`
- `formaction`
- `poster`
- `xlink:href`

地址不得含字节值小于 33 的空白/控制字符或 DEL。若出现显式协议，只允许`http:`、`https:`、`mailto:`和`tel:`；没有显式协议的相对地址、根路径、查询和片段可以使用。

这一校验是保守的第一道边界，不验证主机可信度、重定向目标、同源策略或业务权限。应用仍应避免把不可信地址用于导航、下载或服务端请求。

## 原始内容

`HTML原始内容`和`原始`完全跳过转义。推荐只用于：

- 源码中的固定实体或已审计片段；
- 由另一个具有同等安全保证的结构化渲染器生成的内容；
- 已经过专用、上下文正确的 HTML 清洗器处理的内容。

代码审查时可以搜索`HTML.原始（`或`HTML原始内容（`，集中核查所有信任升级点。

## 非目标

0.1 不提供 HTML 解析/清洗、CSS URL 分析、`style`属性策略、脚本安全、CSP 生成、Trusted Types 或 SVG 子语言校验。对外站点应在 HTTP 层补充 CSP、`X-Content-Type-Options`、合理 Cookie 属性和 HTTPS；可参考[yanxu-webblog 的安全首部中间件](https://github.com/yanxulang/yanxu-webblog/blob/main/src/应用.yx)。

## 回归检查

`tests/安全渲染.yx`覆盖文字与属性转义、危险 URL、控制字符、布尔属性和文档输出。报告安全问题时，请提供最小输入、实际输出、期望阻止的上下文和言序版本。
