# 性能与基准

## 目标

言页优先保证转义、安全策略和资源预算一致，再减少完整正文的额外驻留。`流式渲染`的主要价值是逐片消费和有界上层缓冲，不承诺在每台机器上都比完整联结更快。

## 仓库负载

`benchmarks/安全渲染性能.yx`构造一个完整 HTML5 页面：

- 20 篇`article`；
- 每篇包含标题、正文和链接；
- 覆盖中文、`<`、`>`、`&`和引号转义；
- 同一节点图分别走完整渲染和增量渲染；
- 校验两条路径的总字符数完全一致；
- 拒绝 1 以下、500 以上或非整数轮数。

默认 50 轮。1.0.0 发布候选的确定性校验值为：

| 指标 | 50 轮结果 |
| --- | ---: |
| 文章数 | 20 |
| 完整字符 | 145600 |
| 增量字符 | 145600 |
| 增量片段 | 19050 |

毫秒数取决于 CPU、构建模式和系统负载，不写入兼容承诺。

## 运行源码基准

```sh
yanxu 包 锁 --离线 benchmarks
yanxu --max-steps 10000000 字节 benchmarks/安全渲染性能.yx -- 50
```

快速回归可以使用 5 轮；发布验收使用 50 轮。对性能趋势做统计时，应预热后多次运行并保留机器、系统、言序版本、构建模式和轮数。

## 运行 Release 制品

```sh
yanxu 编 benchmarks --release -o /tmp/yanxu-html-benchmark.yxb
yanxu --max-steps 10000000 行 /tmp/yanxu-html-benchmark.yxb -- 50
```

源码 VM 与 Release YXB 都必须产生相同校验值。若字符数或片段数变化，应先判断这是预期输出契约变化、分片实现变化还是回归，再更新基准；不能只修改期望值让失败消失。

## 成本模型

设访问节点数为 N、全部属性数为 A、输出字符数为 U：

- 节点遍历为 O(N)；
- 属性验证与渲染为 O(A)，重复属性在构造期逐项扫描，单个极端元素最坏为 O(A²)，但属性硬上限为 1024；
- 转义与字符统计为 O(U)；
- 完整渲染保存全部片段并联结，额外空间为 O(U)；
- 流式渲染不在库内保存完整正文，额外空间主要由深度和调用方写入器决定。

字符预算按 Unicode 字符而不是 UTF-8 字节计算。网络服务若需要精确内存或带宽上限，还应在编码后限制字节数。

## 调优

- 对可一次返回的小页面使用`安全渲染`，语义最简单；
- 对大响应使用`流式渲染`，并让上层写入器提供有界队列和背压；
- 用`片段`组合同级节点，避免无意义包装元素；
- 组件构建器不要重复读取网络或数据库；渲染前先完成数据装载；
- 按真实最大页面收紧预算，尽早拒绝异常结构；
- 不用`原始`跳过转义来追求速度，信任边界不能由微基准决定；
- 不依赖当前片段切分数量优化协议，片段边界不是稳定序列化格式。

性能变更必须同时保留安全规格、完整/增量校验和以及 Release 构建验证。
